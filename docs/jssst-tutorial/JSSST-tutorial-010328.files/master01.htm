まず用語の定義です。 Javaは可搬性ポータビリティが特徴のひとつですが、
後で説明する性能の可搬性は満足しないという問題があります。
我々の本研究における提案はDownloadable Self-tuning Potable Libraryです。
これは、Javaの特徴であるPortablityを活かしつつ、
ここで性能の可搬性という言葉を定義します。
ある最適化を施したとき, それが全ての処理系やほかの環境（CPU、OS、言語ランタイム、言語ミドルウェアetc）上で同程度に効果的か否か。
たとえば、ある環境Aでの性能がピーク性能の80％でたとすると、
ほかの環境Bでの性能がピーク性能の80％でたという状況で
性能の可搬性があるといいます。
各環境(プラットフォーム)できちんとした性能が出ることが重要

我々は, 各Java処理系の行列積のピーク性能をシステマティックに得るために, ATLASをJavaに適用できるように変更したATLAS for Javaを開発しました。ATLAS for Javaは主にJavaで記述されている. オリジナルのATLASに対し, 新たに作成した部分は, 図の点線内です. ブロック内コード生成器がxemit.java, 性能のチェック部がfc.javaである. xemit.javaは指定されたパラメータに応じて行列最内周ループ計算を最適化したソースを出力する. 具体的にはATLASと同じアルゴリズムでブロッキング, アンローリング, ソフトウェアパイプライニングを行う最適化されたJavaのソースを生成する. fc.javaはテストルーチンであり, ATLASのxmmsearchによって呼び出され, 行列を格納するための配列を生成し, 必要数のループを行い, 実行時間を測定した後Mflops値を得る. 先に述べたCとJavaの違いにより, ATLAS for Javaの実装では単純なソースの書き換えのみならず, 以下のような変更が必要であった: ･Javaでは配列ポインタがないため, 配列のインデックスと内容を保持するクラスを定義した. Javaではオブジェクトは参照渡しのため, このクラスのインスタンスを渡すことで, 配列全体およびそのインデックスを一度に参照渡しすることが可能となる. ･関連するが, Javaではスカラ値への参照渡しができない. しかし, ATLASでは探索の結果の複数のパラメタを参照渡しで得てさらに返すことをしている. そこで, 変数を参照渡しのためのWrapperクラスを定義した. ･オリジナルのATLASはfloat, double型をサポートしているが,ATLAS for Javaで使用するデータ型はdouble型に制限する.
